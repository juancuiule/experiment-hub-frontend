# Multi-Agent Development Workflow Design

**Date:** 2026-04-22
**Scope:** v1 тАФ bug fixes and 4 draft feature specs (excludes visual flow builder)

---

## Problem

Running a solo development workflow on a project like this means tasks happen sequentially: fix one bug, then the next, then the next. Independent tasks that could run in parallel are serialized by default. Claude Code quota limits compound this тАФ one agent running one task at a time burns through quota without throughput gains.

The goal is a workflow where independent tasks run simultaneously, each in its own terminal pane with a live output stream, while a planner ensures the work is coherent and a reviewer handles merge order.

---

## Architecture

Four components:

### 1. Planner (`scripts/plan.ts`)

Calls Opus 4.7 via the Anthropic API. Receives a task description and the project's file tree. Produces a structured plan as JSON.

Each plan contains a list of subtasks:

```json
{
  "subtasks": [
    {
      "id": "fix-validation",
      "description": "Fix the requiresInteraction refine in lib/validation.ts тАФ the refine always returns true regardless of component type",
      "files": ["lib/validation.ts", "lib/components/response.ts"],
      "model": "haiku",
      "branch": "fix/validation",
      "dependsOn": []
    },
    {
      "id": "fix-conditional-context",
      "description": "Fix Conditional.tsx to use mergeContext instead of spread тАФ spread strips foreachData from the context",
      "files": ["src/components/control/Conditional.tsx", "lib/types.ts"],
      "model": "haiku",
      "branch": "fix/conditional-context",
      "dependsOn": []
    }
  ]
}
```

Fields per subtask:
- `id` тАФ unique slug
- `description` тАФ precise task for the worker agent, including the bug/feature and the why
- `files` тАФ exactly which files are relevant (workers load only these)
- `model` тАФ `haiku` for single-file edits, `sonnet` for refactors, `sonnet` for features
- `branch` тАФ git branch the worker commits to
- `dependsOn` тАФ list of subtask IDs that must finish before this one starts

### 2. tmux Launcher (`scripts/launch.sh`)

Reads the plan JSON. Creates a tmux session with a fixed layout:

- Top pane: shows the plan summary and overall status
- Middle panes: one per subtask (N panes, created dynamically from the plan)
- Bottom pane: idle until all workers finish, then runs the merge reviewer

Each worker pane runs:
```bash
claude --print "$(cat .claude-task/<subtask-id>.md)"
```

Where `.claude-task/<subtask-id>.md` contains the full task prompt generated by the launcher, including: the subtask description, which files to read, which branch to work on, and a commit message template.

### 3. Workers (Claude Code agents)

Each worker is a Claude Code process running in its own tmux pane. Claude Code has full tool access: Read, Edit, Bash, git. No custom worker infrastructure needed.

The task prompt handed to each worker is structured:

```markdown
## Task
Fix the requiresInteraction refine in lib/validation.ts.

The refine function always returns true because it checks `component.props` 
without narrowing the component type first. It should only require interaction 
for response-family components.

## Files to read
- lib/validation.ts
- lib/components/response.ts

## Git
Branch: fix/validation (create from main if it doesn't exist)
Commit message: "fix: requiresInteraction refine always returns true"

## Done when
The refine correctly returns false for content/layout/control components 
and true only for response components that have no defaultValue.
```

Workers commit to their branch when done. If a worker is genuinely blocked (ambiguous requirement, missing context), it writes a question to `.claude-task/<id>.question` and exits. The human reads it in the pane and can restart the worker with clarification added to the prompt.

### 4. Merge Reviewer (`scripts/review-merge.ts`)

Calls Opus 4.7 via API after all workers have committed. Reads the diff of each worker branch against main. Produces:

- Safe merge order (respecting dependencies and minimizing conflicts)
- Conflicts to resolve manually (if any)
- A brief summary of what each branch changes

Output is printed to the bottom tmux pane. Human reviews and merges manually in Cursor/VS Code.

## Data Flow

```
User runs: pnpm plan "Fix validation bugs and remove debug artifacts"
  тЖТ plan.ts calls Opus тЖТ plan.json written to .claude-task/plan.json
  тЖТ task prompts written to .claude-task/<id>.md

User runs: pnpm launch
  тЖТ launch.sh reads plan.json
  тЖТ creates tmux session with N worker panes + status pane + reviewer pane
  тЖТ each worker pane starts: claude --print "$(cat .claude-task/<id>.md)"
  тЖТ workers run in parallel, each on its own git branch
  тЖТ when all workers exit: reviewer pane activates
  тЖТ review-merge.ts calls Opus тЖТ prints merge order

User reviews diffs in Cursor, merges branches in recommended order
```

## Cost Management

| Technique | Applied where | Effect |
|---|---|---|
| Prompt caching | All API calls share system prompt + file tree | ~10├Ч cheaper cache reads |
| Model routing | Haiku for 1-file edits, Sonnet for refactors | 5├Ч cheaper than Sonnet baseline |
| Batch API | Merge reviewer (not time-critical) | 50% discount |
| Context scoping | Plan specifies exact files per subtask | Workers load 2-4 files, not full repo |
| Task budgets | Planner call: `task_budget: {type: "tokens", total: 20000}` | Caps Opus planning spend |

Estimated cost for v1 scope (9 bugs + 4 features): **~$2.50тАУ3.00 per full run**.

## File Structure

```
scripts/
  plan.ts          # Opus planner тАФ takes task description, writes plan.json
  launch.sh        # tmux launcher тАФ reads plan, creates session
  review-merge.ts  # Opus merge reviewer тАФ reads diffs, outputs merge order
  lib/
    anthropic.ts   # Shared API client with caching config
    git.ts         # Git helpers (create branch, get diff, list branches)
    tmux.ts        # tmux session/pane management helpers

.claude-task/      # Generated per-run (gitignored)
  plan.json
  <subtask-id>.md  # Task prompt per worker
  <subtask-id>.question  # Written by worker if blocked
```
